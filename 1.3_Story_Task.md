Here is the 4-person task breakdown for **Sub-Story 1.3 (Tabular Data
Extraction & Fallback)**. This phase bridges the gap between deep AI
document parsing, batch backend processing, and real-time frontend
notifications.

**Person 1: AI / Data Engineer (Extraction Engine)**

**Focus:** Sub-Story 1.3a & 1.3b --- Table detection, raw data
preservation, JSON structuring, and exception generation for flat
images.

- **Tools to Use:** Python, specialized table extraction libraries
  (e.g., Camelot, Tabula-py, or pdfplumber), LlamaIndex/LLMs (for
  complex table structures), and Pydantic.

- **How to Build:**

  - **Table Detection & Parsing:** Build the core extract_tables(pdf,
    source_doc_id) function to scan classified PDFs for grid structures
    and extract rows and columns.

  - **Strict Format Preservation:** Enforce strict rules to extract data
    exactly as it appears. Prevent the AI from normalizing math, merging
    adjacent columns, or stripping units (e.g., \"\$1.5M\" must remain
    exactly \"\$1.5M\", not converted to 1500000).

  - **JSON Construction:** Map the raw extracted data into the exact
    required JSON contract: {tables: \[{rows, columns, source_doc_id,
    page_number}\]}.

  - **Failure Detection (Unhappy Path):** Implement logic to detect if a
    PDF is a flattened image (lacking a text layer) or contains
    corrupted tables. When detected, explicitly raise
    an ExtractionFailureException to signal the orchestrator.

**Person 2: Backend Engineer (FastAPI Batch Orchestration)**

**Focus:** Sub-Story 1.3a & 1.3b --- Batch loop execution, database
persistence, and graceful error isolation.

- **Tools to Use:** Python, FastAPI, SQLAlchemy (or SQLModel), and an
  internal event publisher (e.g., Redis Pub/Sub or direct HTTP client).

- **How to Build:**

  - **The Batch Loop:** Write a processing loop that iterates over all
    classified financial PDFs in the batch, feeding each one to Person
    1\'s extraction engine.

  - **Happy Path Persistence:** When extraction succeeds, parse the
    returned JSON and execute an INSERT into the
    PostgreSQL extracted_metrics table for each metric row. Update the
    overall document record status to Extracted.

  - **Graceful Isolation (Unhappy Path):** Wrap the extraction call in
    a try/except block. If the ExtractionFailureException is caught,
    safely halt processing for *that specific document only*.

  - **State & Signaling:** For failed documents, execute an UPDATE on
    the database record setting the status
    to Extraction_Failed_Requires_OCR. Immediately emit an event
    downstream (to the Gateway) containing {doc_id, status} before the
    loop automatically continues to the next PDF in the batch.

**Person 3: Database & Gateway Engineer (PostgreSQL & Spring Boot)**

**Focus:** Sub-Story 1.3a & 1.3b --- Schema execution and real-time
frontend event routing.

- **Tools to Use:** PostgreSQL, Java/Kotlin, Spring Boot, and Spring
  WebSockets or Server-Sent Events (SSE).

- **How to Build:**

  - **Metrics Schema:** Execute the database migration to create
    the extracted_metrics table. Ensure columns strictly support the
    required metadata: metric_name, raw_value, unit, source_doc_id,
    and page_number.

  - **Real-Time Bridge:** Create a real-time messaging pipeline in the
    Spring Boot Gateway. When FastAPI broadcasts an extraction failure,
    the Gateway must catch this internal event and forward it to the
    React frontend.

  - **Session Management:** Ensure the Gateway routes the failure event
    only to the specific authenticated investment analyst who initiated
    the upload, rather than broadcasting it to every user on the
    platform.

**Person 4: Frontend Engineer (React UI)**

**Focus:** Sub-Story 1.3b --- Real-time event consumption and dynamic
user notifications.

- **Tools to Use:** React, TypeScript, Context API / Zustand (for global
  state), and a WebSocket or SSE client library.

- **How to Build:**

  - **Real-Time Subscription:** Implement a connection in the React app
    that listens to the Spring Boot Gateway\'s real-time feed while the
    ingestion batch is processing.

  - **Dynamic Banners:** When the client receives a message payload
    matching Extraction_Failed_Requires_OCR, dynamically dispatch a UI
    notification.

  - **User Experience:** Render a human-readable banner (e.g., *\"OCR
    review required for InvestCo_Report.pdf\"*) that alerts the analyst
    without interrupting their current workflow or requiring them to
    refresh the page.

**Critical Files for Sub-Story 1.3 Implementation:**

- **table_extractor.py (Person 1):** Implements the logic to ensure that
  adjacent columns remain distinct and that original numerical formats
  (e.g., shorthand or specific currency symbols) are never altered
  during the conversion to JSON\[cite: 1\].

- **async_pipeline.py (Person 2):** Modified to wrap each extraction in
  a try-except block; it ensures that a flat-image PDF lacking a text
  layer only halts extraction for that specific document while the rest
  of the batch continues\[cite: 1\].

- **NotificationBridge.java (Person 3):** Serves as the orchestration
  point that catches the Extraction_Failed_Requires_OCR status update
  from the database/FastAPI and pushes it to the correct user
  session\[cite: 1\].

- **ExtractionToast.tsx (Person 4):** A React component that listens for
  the specific \"Requires OCR\" event and renders a non-blocking
  notification banner for the analyst\[cite: 1\].

- **03-extracted-metrics-schema.sql (Infrastructure):** Initializes
  the extracted_metrics table with the
  mandatory source_doc_id and page_number metadata to support the audit
  chain\[cite: 1\].
