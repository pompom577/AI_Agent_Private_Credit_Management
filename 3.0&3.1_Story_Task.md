Person 1: Backend Engineer (Gateway Middleware & Interception)
Focus: Sub-Story 3.1a & 3.1b — Request interception, payload extraction, early 202
responses, and client timeout detection.
Tools to Use: Java/Kotlin, Spring Boot, Spring Cloud Gateway
(or OncePerRequestFilter), and Servlet/WebFlux connection listeners.
How to Build:
• Targeted Interceptor: Construct a dedicated Gateway Filter designed
specifically to listen for traffic destined for flagged "High Risk" endpoints
(e.g., /api/credit/approve).
• Payload Parking Mechanism: Read the incoming request body, headers,
target URL, and agent_id from the AI agent. Completely halt the routing chain
so the request never reaches its intended destination. Pass this data to the
database service to be persisted.
• Agent Acknowledgment: Construct and return a 202 Accepted (Pending) HTTP
response back to the originating AI agent, signaling that the action is safely
held for human review.
• Timeout Catch (Unhappy Path): Implement a connection state listener. If the
AI agent terminates the network connection before the Gateway can deliver
the 202 response, intercept the resulting ClientAbortException. Trigger an
immediate database update changing the parked payload's status
to Delivery_Timeout_Warning.
Person 2: Database Engineer (Persistence & Core API)
Focus: Sub-Story 3.1a, 3.1b, & 3.2a — Quarantine schema creation, state machine
updates, and exact payload forwarding logic.
Tools to Use: PostgreSQL, Spring Data JPA / Hibernate, Jackson JSON, and a
Java HTTP client (e.g., RestTemplate or WebClient).
How to Build:
• Quarantine Schema: Execute DDL migrations to create the
PostgreSQL quarantine table containing columns
for quarantine_id, endpoint, payload (JSONB), agent_id, and status.
• State Management Endpoints: Build the REST controllers that the React
frontend will consume: GET /quarantine, POST /quarantine/{id}/approve, and POST
/quarantine/{id}/reject.
• Discard Logic: When the /reject endpoint is hit, update the specific database
row's status to Discarded and permanently remove it from the active review
queue.
• Execution Logic: When the /approve endpoint is hit, query the exact original
JSON payload and target endpointfrom the database. Use an internal HTTP
client to fire that precise payload directly to its intended destination, bypassing
the middleware filter. Update the status to Approved upon successful execution.

Person 3: Frontend Engineer (HITL Dashboard UI)
Focus: Sub-Story 3.1a & 3.2a — Compliance officer interface, real-time queueing,
and approval/rejection workflows.
Tools to Use: React, TypeScript, a state management library (Context
API/Zustand), and WebSocket/SSE hooks.
How to Build:
• Real-Time Queue Dashboard: Build a secure view for Compliance Officers
that displays a live list of parked payloads. Connect this view to the real-time
notification stream so new interceptions appear instantly without a page
refresh.
• Timeout Alert Badges: Implement conditional UI rendering to prominently
flag any payload carrying the Delivery_Timeout_Warning status. Highlight these in
red to alert the officer that the autonomous agent might attempt to duplicate
the action.
• Payload Review Modal: Create a detail view where the officer can inspect
the exact JSON payload the AI agent attempted to send, alongside the target
endpoint and agent ID.
• Action Execution: Implement "Approve" and "Reject" buttons. Attach
confirmation dialogue boxes to both actions to prevent accidental clicks. Wire
these buttons to Person 2's backend POST endpoints and optimistically
remove the item from the local UI queue upon a 200 OK response.
Person 4: DevOps & Security Engineer
Focus: Sub-Story 3.1b & 3.2a — Network timeouts, alerting infrastructure, and
internal routing security.
Tools to Use: Terraform, Docker/Kubernetes, API Gateway configurations, and
Webhook integrations (e.g., Slack, Microsoft Teams, or PagerDuty).
How to Build:
• Timeout Simulation & Enforcement: Configure strict connection read/write
timeouts at the ingress layer (e.g., NGINX or AWS ALB) to accurately mimic
and test network drops between the AI agents and the Spring Boot Gateway.
• Active Officer Alerting: Configure an external notification bridge. When a
payload receives a Delivery_Timeout_Warning status, push an automated alert to
the Compliance Officer's Slack or email channel to ensure immediate visibility
of the unstable agent.
• Internal Routing Security: Establish network policies ensuring that the
internal destination endpoints (the high-risk targets) strictly accept
traffic only from the Gateway's execution service (Sub-Story 3.2a) and
explicitly block any direct network attempts from the AI agents, enforcing the
HITL architecture at the infrastructure level.

Here is a comprehensive test suite designed to validate the Human-in-the-Loop
(HITL) Architecture (Sub-Stories 3.1 & 3.2). This suite guarantees that high-risk
payloads are successfully quarantined, network disconnections are handled
gracefully, and compliance officers have full deterministic control over payload
execution.
