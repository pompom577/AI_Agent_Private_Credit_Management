Person 1: Frontend Engineer (PDF Rendering & Overlays)
Focus: Sub-Story 2.2a — Split-screen layout, PDF parsing in the browser, and
coordinate highlight rendering.
• Tools to Use: React, TypeScript, a robust PDF library (e.g., react-pdf or
Mozilla's pdf.js), and CSS/Canvas for absolute positioning.
• How to Build:
o Split-Screen Container: Build the core UI layout that allows the digital
term sheet to compress to the left half of the screen while opening a
dedicated document viewing panel on the right.
o PDF Viewer Implementation: Integrate the PDF library to accept the
binary stream (or Blob URL) from the backend and accurately render
the specific page_number requested.
o Coordinate Highlight Engine: Build a translation function that takes
the backend's spatial coordinates (x_min, y_min, x_max, y_max) and maps
them to the actual rendered pixel dimensions of the PDF canvas. Draw
a visual highlight box over the exact target cell and trigger an automatic
scroll to bring that box into the center of the user's viewport.
Person 2: Frontend UX Engineer (State Management)
Focus: Sub-Story 2.2b — Loading states, click-spam prevention, and graceful error
boundaries.
• Tools to Use: React Hooks (useState, useRef), Tailwind CSS / styled-
components (for skeleton animations).
• How to Build:
o Spam Prevention: Implement state logic on the metric row component
so that when a user clicks a metric, its onClick handler is immediately
disabled to prevent duplicate network requests while the panel is
opening.
o Skeleton Loaders: Build a shimmering placeholder component
(<SkeletonLoader />) that visually mimics a document page. Render this
instantly in the right-hand panel the moment the user clicks, bridging
the gap during the network fetch.
o Error & Retry Handling: Intercept HTTP 504 (Gateway Timeout) or
network failure errors. When caught, automatically swap the skeleton
loader out for a clear error state (e.g., "Failed to load document")
featuring a "Retry" button. Ensure the onClick lock is released on both
success and failure states.
Person 3: Backend API Engineer (Spring Boot)
Focus: Sub-Story 2.2a & 2.2b — Document routing, byte streaming, and strict
timeout configurations.
• Tools to Use: Java/Kotlin, Spring Boot, Spring WebFlux (for reactive
streaming) or standard MVC StreamingResponseBody, and Cloud Storage SDKs
(AWS S3, Azure Blob).

• How to Build:
o Retrieval Endpoint: Create the GET
/documents/{source_doc_id}/page/{page_number} REST endpoint to serve the
specific document asset to the frontend.
o Efficient Byte Streaming: Rather than downloading a massive 500
MB PDF into the backend's server memory, implement a stream-
through mechanism. Fetch the file byte stream from the secure cloud
bucket and pipe it directly to the HTTP response output stream to
minimize memory overhead and time-to-first-byte (TTFB).
o Timeout Guardrails: Configure strict timeout bounds on the endpoint.
If the storage bucket fails to respond or the stream stalls beyond the
required SLA limits, cleanly abort the connection and return a 504
Gateway Timeout to trigger the frontend's error state.
Person 4: Infrastructure & Performance Engineer
Focus: Sub-Story 2.2a — Caching layers, bucket optimization, and guaranteeing the
< 2-second SLA.
• Tools to Use: Terraform / Infrastructure as Code, Redis (for Gateway
caching), or a CDN (CloudFront/Cloudflare).
• How to Build:
o SLA Verification: Test the raw retrieval speed from the secure bucket
to the Gateway to the Client. If the raw fetch of a large PDF exceeds
the 2-second SLA required by the Investment Committee, implement
an optimization layer.
o Caching Strategy: Provision a Redis instance or an internal caching
layer for the Spring Boot application. Since Deal Analysts will likely
click multiple metrics originating from the same source document,
cache the active PDF bytes in fast memory during the session to
instantly serve subsequent page requests.
o Page Splitting Optimization (Optional): If full-document streaming is
too slow, deploy a lightweight serverless function (e.g., AWS Lambda)
triggered on the initial ingestion (Sub-Story 1.1) that proactively bursts
large PDFs into single-page assets in the bucket, allowing the Gateway
to fetch only the exact 200 KB page rather than navigating a 100 MB
monolithic file.
