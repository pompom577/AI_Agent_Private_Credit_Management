Here is the 4-person detailed task breakdown to implement Sub-Story 3.3a (Basic
Audit Logging) based on your system diagrams. This phase focuses entirely on
immutability, transactional guarantees, and operational alerting to ensure every
human decision is permanently recorded.
Person 1: Database Engineer (Audit Schema & Constraints)
Focus: Sub-Story 3.3a — Append-only ledger creation and database-level security
constraints.
• Tools to Use: PostgreSQL, Database Migration Tools (Flyway or Liquibase),
and SQL/DDL.
• How to Build:
o Ledger Schema: Create the hitl_audit_ledger table with precise
columns: action_url, timestamp(TIMESTAMPTZ), decision (Enum/String:
Approve/Reject), officer_id (UUID), and quarantine_id (UUID).
o Append-Only Enforcement: Just like the metric audit table in Sub-
Story 2.0, write explicit database-level triggers and constraints
on hitl_audit_ledger that immediately raise exceptions if
an UPDATE or DELETEoperation is attempted.
o Permission Revocation: Revoke UPDATE and DELETE privileges for
the standard application database role on this specific table to
guarantee cryptographic-like immutability at the infrastructure level.
Person 2: Backend API Engineer (Spring Boot Audit Logic)
Focus: Sub-Story 3.3a — Decision composition, database insertion, and
transactional boundaries.
• Tools to Use: Java/Kotlin, Spring Boot, Spring Data JPA / Hibernate, and
Spring Transaction Management (@Transactional).
• How to Build:
o Audit Composition: Inside the existing /approve and /reject endpoint
controllers (built in 3.2), instantiate a new audit record object containing
the {action_url, timestamp, decision, officer_id, quarantine_id}immediately after
the payload decision is processed.
o Transactional Guarantee: Wrap the payload release/discard logic and
the hitl_audit_ledger insertion within a strict,
single @Transactional boundary.
o Failure Catching: Ensure that if the INSERT into the audit ledger fails
for any reason (e.g., database connection loss), the entire transaction
rolls back. The system must not release a quarantined payload if it
cannot simultaneously write the audit record.
Person 3: Frontend Engineer (HITL Dashboard UI)
Focus: Sub-Story 3.3a — User feedback loops and confirmation states.
• Tools to Use: React, TypeScript, Axios/Fetch, and a Toast notification library
(e.g., react-toastify or sonner).

• How to Build:
o Network Acknowledgment: Update the onClick handlers for the
"Approve" and "Reject" buttons to strictly wait for the 200 OK response
from the Spring Boot API before mutating the local UI state.
o Success Toast: Upon receiving the 200 OK, render a specific
confirmation toast (e.g., "Decision confirmed and permanently audited")
to assure the Compliance Officer that their action was logged securely.
o Failure Handling: If the API returns a 500 Internal Server Error (because
the audit write failed), leave the payload in the active queue and
display an error toast (e.g., "Action failed: Audit ledger unreachable.
Please try again.").
Person 4: DevOps / Site Reliability Engineer (SRE)
Focus: Sub-Story 3.3a — Operational monitoring and failed transaction alerting.
• Tools to Use: Prometheus, Grafana, ELK Stack / Datadog, and PagerDuty
(or equivalent alerting bridge).
• How to Build:
o Write Failure Alerts: Configure monitoring on the Spring Boot API
logs and database transaction metrics. Set up an immediate alert to the
Ops/SRE team if a transaction fails specifically due to a write error on
the hitl_audit_ledger.
o High-Severity Paging: Because a failure to write to the audit ledger
effectively freezes the Compliance Officer's ability to approve or
reject any deals, configure this alert to trigger a high-severity page, as
the ledger write must succeed before any decision is considered final.

compliance failures.
